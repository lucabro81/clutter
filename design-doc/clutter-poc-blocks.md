# CLUTTER — POC Building Blocks

**Version**: 0.1.0-draft
**Status**: Exploration
**Author**: Luca

---

## Approach

Custom compiler/transpiler. `.clutter` files → AST → semantic validation → target code.

Architectural references: Babel, `@vue/compiler-sfc`, Svelte compiler. The structure is well-established — minor variations between projects, dependency order between blocks always the same.

---

## Blocks in dependency order

### 1. Lexer

First contact with the raw text of the `.clutter` file. Transforms the character sequence into meaningful tokens.

Token examples: `<`, `Column`, `gap`, `=`, `"md"`, `>`, `{`, `title`, `}`, `---`

Everything else in the pipeline depends on this block. No other block can exist without it.

---

### 2. Parser

Takes the tokens produced by the lexer and builds an AST (Abstract Syntax Tree).

Responsibilities:
- Recognise the `---` separator and distinguish the two sections of the file (logic / template)
- Reconstruct the hierarchical structure of the template (component nesting)
- Recognise props, values, variable references
- Treat the logic section as an opaque TypeScript block

Output: tree representing the file in a form that subsequent blocks can manipulate.

---

### 3. Semantic Analyzer

Traverses the AST and validates meaning — not syntax. This is the block that demonstrates the project's primary value.

Responsibilities:
- Load `tokens.clutter` (design system)
- Verify that every prop receives a value present in the tokens
- Verify that variable references in the template section exist in the logic section
- Produce human-readable semantic errors

Example expected error:
```
Error [CLT001] — line 4, column 12
Value 'xl2' does not exist for prop 'gap'.
Valid values: xs, sm, md, lg, xl, xxl
```

---

### 4. Code Generator

Traverses the validated AST and produces the target code. Separate from the analyzer because targets are interchangeable.

Targets for the POC: Vue SFC or static HTML.

Responsibilities:
- Map each AST node to the corresponding construct in the target
- Preserve the logic section (TypeScript) unchanged or with minimal modifications
- Correctly reference the runtime components

---

### 5. Runtime (minimal)

Concrete implementation of the built-in components. The code generated by the code generator references them — they must exist somewhere.

Components for the POC: `Column`, `Row`, `Box`, `Text`, `Button`

For the POC these can be simplified implementations. What matters is that they run and produce visually correct output. They are not exposed to the developer — they are opaque.

---

### 6. CLI

The glue for the entire pipeline. Takes a `.clutter` file, passes it through blocks 1→4, writes the output to disk.

Minimal interface for the POC:
```
clutter build <file>
```

Nothing else. No watch mode, no configuration, no debug output by default.

---

## Dependency diagram

```
tokens.clutter
      |
      ↓
.clutter file
      |
      ↓
   [Lexer]
      |
      ↓
   [Parser] → AST
      |
      ↓
[Semantic Analyzer] ← tokens.clutter
      |
      ↓
[Code Generator]
      |
      ↓
  Target output
  (Vue SFC | HTML)
      |
      ↓
   [Runtime]
   (referenced by the output)

   [CLI]
   (orchestrates everything)
```

---

## Notes

- The block order is constrained by dependencies — it is not arbitrary
- CLI and Runtime come last because they are consumers of everything else
- The Semantic Analyzer is the block with the most demonstrative value for the POC — it is where you see that "a wrong value does not compile"
- LSP, editor tooling, plugin architecture, multiple targets: out of scope for the POC
