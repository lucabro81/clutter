## What the CLI is and why it exists

The CLI (Command Line Interface) is the public entry point of the compiler — the only thing the developer sees and uses directly. Everything else in the pipeline (Lexer, Parser, Semantic Analyzer, Code Generator) is invisible: the CLI orchestrates them in sequence and presents the result.

It is the simplest block to implement, but it is the one that determines the perceived quality of the entire tool. A technically perfect compiler with a confusing or verbose CLI is used reluctantly.

---

## Interface for the POC

For the POC, the CLI exposes a single command:

```
clutter build <file> [--target <vue|html>]
```

**Arguments:**

| Argument | Required | Default | Description |
|---|---|---|---|
| `<file>` | Yes | — | Path to the `.clutter` file to compile |
| `--target` | No | `vue` | Compilation target |
| `--out` | No | same directory as source | Output directory |

**Examples:**

```bash
# Compile to Vue SFC (default)
clutter build src/components/Card.clutter

# Compile to HTML
clutter build src/components/Card.clutter --target html

# Specify output directory
clutter build src/components/Card.clutter --out dist/
```

---

## Execution flow

When the developer runs `clutter build`, the CLI:

```
1. Reads and validates arguments
2. Verifies that the source file exists
3. Searches for tokens.clutter (from the file's directory, walking up to the root)
4. Reads the source file
5. Invokes the Lexer → array of tokens
6. Invokes the Parser → AST
7. Invokes the Semantic Analyzer → validated AST or error list
8. If there are errors → prints errors, exits with code 1
9. If no errors → invokes the Code Generator → output string
10. Writes the output file to disk
11. Prints success message, exits with code 0
```

The exit code (`0` = success, `1` = error) is important for CI/CD integration and automated scripts.

## Screen output

### Success

```
✓ Card.clutter → Card.vue (12ms)
```

Concise. Nothing more.

### Errors

```
✗ Card.clutter — 2 errors

  Error [CLT102] line 4, col 12
  Value 'xl2' is not valid for prop 'gap' of 'Column'.
  Valid values: xs, sm, md, lg, xl, xxl

    4 │ <Column gap="xl2">
                    ^^^

  Error [CLT104] line 9, col 8
  '{foo}' is not declared in the logic section.

    9 │   <Text>{foo}</Text>
                ^^^^^^
```

Each error shows:
- Code and position
- Human-readable message
- Hint with valid values when applicable
- The source fragment with the exact point highlighted

The source fragment with highlighting is the difference between a professional compiler and an amateur one. In Rust, `miette` produces this output automatically from the `ErrorNode` objects with positions that the Semantic Analyzer already produces.

## Searching for tokens.clutter

The `tokens.clutter` file is not passed explicitly — the CLI searches for it automatically by walking up the directory structure from the source file's location, until reaching the filesystem root or finding it.

```
src/
  components/
    Card.clutter     ← source file
  tokens.clutter     ← found here
```

This behaviour is identical to how `eslint`, `prettier`, and `tsconfig.json` are searched — familiar to any JavaScript developer.

If `tokens.clutter` is not found, the CLI produces an explicit error:

```
✗ tokens.clutter not found.
  Searched in: src/components/, src/, ./
  Create a tokens.clutter file in the project root.
```

## Future commands (out of scope for the POC)

For completeness, the interface at scale will include:

```bash
# Check without producing output
clutter check <file>

# Compile all .clutter files in a directory
clutter build src/ --out dist/

# File watcher (re-runs on change)
clutter watch src/

# Validate tokens.clutter
clutter tokens validate

# Show version
clutter --version
```

None of these are needed for the POC.

## Stack: clap

In Rust, the CLI is built with `clap` — the de facto standard library for argument parsing. It automatically handles:
- Argument and flag parsing
- Help generation (`clutter --help`)
- Value validation (e.g. `--target` only accepts `vue` or `html`)
- Error messages for missing or invalid arguments

```
clutter build

Usage: clutter build <file> [OPTIONS]

Arguments:
  <file>  .clutter file to compile

Options:
  --target <TARGET>  Compilation target [default: vue] [possible values: vue, html]
  --out <DIR>        Output directory
  -h, --help         Print help
```

This output is generated by `clap` without writing a single line of manual code.

---

## Block input and output

**Input**: command-line arguments

**Output**:
- File written to disk (success)
- Screen messages (errors or confirmation)
- Process exit code (`0` or `1`)

The CLI is the only block that has side effects visible to the outside — all other blocks receive data and return data, without directly touching the filesystem or stdout.
